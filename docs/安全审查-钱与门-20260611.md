## 发现与修复

### P1-1 IP 限频可被伪造头绕过(已修)
- 位置:server.js `app.set("trust proxy", true)`
- 攻击:nginx 追加模式传 X-Forwarded-For,Express trust-all 取最左值=请求方自填。攻击者每次换个假 IP,短信 20 次/小时/IP、下单 5 次/分/IP 的限频全部失效 → 短信按条计费被刷钱、垃圾订单灌库。手机号层限频(60s/条)仍在,所以是"成本放大"不是"无限轰炸单人"。
- 修复:`trust proxy = 1`(只剥本机 nginx 一跳,req.ip=真实来源)。

### P1-2 验证码用 Math.random 生成(已修)
- 位置:server.js 发码逻辑。Math.random 是可推算的伪随机(V8 状态可恢复是公开研究),理论上攻击者采样自己手机的验证码可缩小他人验证码搜索空间。
- 修复:`crypto.randomInt(100000, 1000000)`(密码学安全随机源)。

### P2-1 回调不核对实付金额(已加固)
- 攻击面:签名已保真,微信不会发错金额;但商户配置事故/串单场景下会按订单面值入账。已加:解密载荷 `amount.total` ≠ 订单 `amount_cents` → 拒绝入账并报错(可见性优先,返 500 让微信重试出告警痕迹)。

### P2-2 回调无时间戳新鲜度检查(已加固)
- 攻击面:截获的合法回调可无限期重放;幂等索引挡住了实际危害(不会重复开通),但规范要求 ±5 分钟窗口。已加:超窗直接拒(`stale-timestamp`)。

### P2-3 生产环境接受 FAKE_CODE(已修)
- 位置:lib/wechat-oauth.js。原逻辑 env 齐全时仍接受 `FAKE_CODE_` 前缀 → 已登录用户可手动调 callback 给自己 session 塞假 openid(危害低:下单时微信会拒假 openid,无资金损失)。已改:env 齐全时抛错拒绝。

### 文档修正
- lib/session.js 注释原说"生产应把 cookie path 收敛到域前缀"——与跨子路径共享设计矛盾(收窄后 A600/A800 收不到 cookie),已改为说明 path=/ 是必然要求、隔离靠密钥。

## 已验证通过项(攻击者视角逐路径推演)

1. **验签用原始 bytes**:notify 路由单独挂 express.raw,签名串 `timestamp\nnonce\nbody\n` 符合 V3 规范,RSA-SHA256 + 公钥 ID 比对齐全
2. **AES-256-GCM 解密**:authTag 分离、AAD 设置正确,篡改密文必抛错
3. **充值幂等**:`membership_ledger.related_order_id` partial UNIQUE 索引兜底;better-sqlite3 同步执行,进程内无竞态窗口;极端竞态下第二笔 INSERT 抛错→微信重试→幂等命中,自愈
4. **下载 token**:HMAC-SHA256 + timingSafeEqual(防时序侧信道)+ scope/ref/exp 全校验,10 分钟过期
5. **开放重定向防御**:isSafeFromPath 拒 //协议相对、外域、换行注入;OAuth state 走加密 session 防 CSRF
6. **验证码存储**:bcrypt hash 落库(库泄露不泄码)、5 分钟过期、单码 5 次错误锁、验码 10 次/10 分钟限频、用后即焚(used=1)
7. **查单/订单接口**:按 session.phone 隔离,无横向越权
8. **mock-paid**:有归属校验(payer_phone=session.phone)+ 生产开关关闭双保险
